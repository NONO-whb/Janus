## 一、组织架构图

```
用户
  ↓
首辅 (L1) —— 个人助手功能(待办/提醒) + 业务翻译
  ↓
CEO (L2) —— 项目管家、任务分发、进度记录
  ↓
├──────────┬──────────┬──────────┐
↓          ↓          ↓          ↓
工程部    信息部     资料部    后勤部
(L3)      (L3)       (L3)      (跨域)
(套定额)  (查信息)   (做资料)  (协调)
```

## 二、团队成员完整档案

---

### 【首辅】

**基本信息**
- 层级: L1
- 原名: InteractionAgent
- 性质: 薄客户端，只做翻译和简单助理

**核心职责**
1. 接收用户输入
2. 判断模式：助手模式 vs 业务翻译模式
3. 助手模式：处理待办、提醒、简单对话
4. 业务模式：翻译用户请求为结构化数据

**输入**
```python
{
    "user_message": str,           # 用户原始输入
    "context": List[AgentMessage]   # 对话历史（可选）
}
```

**输出 - 助手模式**
```json
{
    "mode": "assistant",
    "action": "todo|reminder|chat",
    "content": "回复内容",
    "timestamp": "2026-03-28T14:00:00"
}
```

**输出 - 业务翻译模式**
```json
{
    "mode": "business",
    "raw_request": "用户原始输入",
    "extracted_keywords": ["混凝土基础", "定额", "价格"],
    "is_followup": false,
    "conversation_context": ["历史消息1", "历史消息2"],
    "timestamp": "2026-03-28T14:00:00"
}
```

**助手模式触发词**
- "记一下" → 添加待办
- "提醒我" / "闹钟" → 设置提醒
- "待办" / "记着" / "记得" → 待办相关

**异常处理**
| 场景 | 处理方式 |
|------|---------|
| 空输入 | 友好提示"请问有什么可以帮您？" |
| 无法识别 | 默认走业务模式，传给CEO判断 |
| 待办存储失败 | 提示用户，但不阻断流程 |

### 【CEO】

**基本信息**
- 层级: L2
- 原名: PlanningAgent
- 性质: 厚层，项目大脑

**核心职责**
1. 项目设立 - 创建项目目录结构
2. 初始化项目状态
3. 理解用户需求，识别需要哪些部门参与
4. 任务分发给各部门经理
5. 收集各部门进度
6. 维护项目状态（详细时间线）
7. 多轮对话关联识别
8. 意图澄清（信息不足时主动询问）

**输入 - 首辅输出**
```json
{
    "mode": "business",
    "raw_request": "分析学校综合楼项目",
    "extracted_keywords": ["学校综合楼", "分析"],
    "is_followup": false
}
```

**处理流程**
```
1. 检查是否是新项目
   ├─ 是新项目 → 创建项目目录
   │             初始化 PROJECT-STATUS.yaml
   │             记录 timeline: 项目设立
   │
   └─ 是现有项目 → 加载项目状态

2. 分析需求，识别部门
   ├─ "套定额" → 需要工程部
   ├─ "查价格" → 需要信息部
   ├─ "做报告" → 需要资料部
   └─ 多部门 → 全部派发

3. 任务分发
   - 准备项目上下文
   - 调用各部门经理

4. 等待结果，更新状态
   - 记录各部门返回结果
   - 更新进度百分比
   - 记录 timeline

5. 汇总返回
```

**项目状态文件格式 (PROJECT-STATUS.yaml)**

**异常处理**
| 场景 | 处理方式 |
|------|---------|

| 部门返回失败 | 记录失败原因，尝试重试或转人工 |
| 多部门冲突 | 转后勤部协调 |

### 【工程部经理】

**基本信息**
- 层级: L3
- 原名: QuotaDomainManager
- 性质: 造价域大脑，知识管理者

**核心职责**
1. 管理定额知识库
2. 指挥4个专员
3. 监督各环节质量
4. 最终复核结果
5. 知识分类、提炼、学习

**知识库结构**
```
工程部知识库/
├── 规范标准/
│   ├── 建设工程工程量清单计价规范.json
│   ├── 各专业的工程量计算规则/
│   └── 提取的处理规则（如"遇到X按Y处理"）
├── 定额库/
│   ├── 建筑分册/
│   ├── 安装分册/
│   ├── 市政分册/
│   └── 检索索引
├── 专业书籍/
│   ├── 造价师教材/
│   ├── 实战技巧/
│   └── 提炼的技能要点
└── 历史项目/
    ├── 人材机比例参考/
    └── 典型案例/
```

**工作流程**
```
CEO派发任务
    ↓
启动并行任务：
    ├─ 图纸专员 → 分析图纸
    └─ 信息专员 → 分析合同招标文件
    ↓
等待两份PDF输出
    ↓
启动清单整理员
    （输入：原始清单 + 图纸摘要 + 项目条件）
    ↓
等待标准化清单
    ↓
启动套定额员
    （输入：标准化清单）
    ↓
等待定额结果
    ↓
经理复核
    - 检查未匹配项
    - 检查单价合理性
    - 检查换算说明
    ↓
输出结果给CEO
```

**输出格式**
```json
{
    "success": true,
    "content": "工程量清单汇总...",
    "confidence": 85,

        "specialty": "建筑",
```

### 【清单整理员】

**基本信息**
- 上级: 工程部经理
- 职责: 整理标准化清单

**输入**
- 原始清单（Word/Excel/PDF）
- 《图纸要点摘要》PDF
- 《项目条件摘要》PDF

**处理流程**
```
1. 格式转换
   Word/PDF表格 → Excel

2. 表格清洗
   - 取消合并单元格
   - 删除空行
   - 列不允许删除

3. 纠错检查
   - 检查单位列和工程量列是否错位
   - 错别字纠正（"混泥土"→"混凝土"）
   - 单位统一（"立方米"→"m³"）

4. 结构识别
   - 没单位+没工程量 → 标记"分部"（分部分项行）
   - 有单位+有工程量 → 标记"清"（清单项）
   - 非标准12位编码 → 标记"补"（补充清单）

5. 归属划分
   - 从分部A开始 → 到下一个分部B为止
   - 中间所有清单项都属于分部A

6. 重复项汇总
   - 名称+项目特征+单位 三者相同
   → 合并为一行，工程量累加

7. 特征整理
   - 拆分参数合并（如"C30"+"商品混凝土"）
   - 按计价标准修改特征描述
   - 缺失特征 → 标记"待确认"

8. 信息融合
   - 根据图纸摘要补充构造做法
   - 根据项目条件标记甲供材、暂估价
```

**输出 - 标准化清单**
```yaml
标准化清单:
  元数据:
    项目名称: XX学校综合楼
    整理时间: 2026-03-28T14:10:00
    整理人: 清单整理员
    状态: 待确认(3项特征缺失)

  清单项:
    - 行标识: 分部
      序号:
      名称: 第一章 土石方工程

    - 行标识: 清
      序号: 1
      清单编码: "010101001001"
      项目名称: 平整场地
      项目特征: 1.土壤类别:二类土 2.弃土运距:5km
      单位: m²
      工程量: 1200
      来源: 原始清单A1
      特殊标记: []

    - 行标识: 清
      序号: 2
      清单编码: "010101003001"
      项目名称: 挖沟槽土方
      项目特征: 待确认  # 土壤类别未明确
      单位: m³
      工程量: 450
      问题反馈: 图纸未明确土壤类别，需确认

      序号:
      名称: 第二章 砌筑工程

    - 行标识: 补
      序号: 10
      清单编码: "B-001"
      项目名称: 特殊防护
      项目特征: 图纸第15页详图
```

**异常上报**
| 异常 | 处理方式 |
|------|---------|
| 图纸与清单矛盾 | 标记冲突，上报经理 |
| 特征缺失无法推断 | 标记"待确认"，暂停该项 |
| 发现漏项 | 建议添加，上报经理确认 |

### 【套定额员】

**基本信息**
- 上级: 工程部经理
- 职责: 匹配定额、计算合价

**输入**
- 标准化清单（清单整理员输出）

**处理流程**
```
1. 逐行处理清单项

2. 定额匹配
   - 按清单名称+特征匹配定额库
   - 输出定额编号、定额名称

3. 单位换算
   - 清单量 ÷ 定额单位 = 工程量
   - 例：清单10m，定额单位10m → 工程量1
   - 换不了 → 工程量留空

4. 单价查询
   - 查定额基价
   - 拆分人材机

5. 合价计算
   - 工程量 × 基价 = 合价

6. 记录匹配理由
```

**输出格式**
```yaml
定额结果:
  - 行标识: 清
    序号: 1
    清单编码: "010101001001"
    项目名称: 平整场地
    单位: m²
    工程量: 1200

  - 行标识: 定
    序号: 1.1
    清单编码: "A1-1"
    项目名称: 平整场地 机械
    单位: 100m²
    工程量: 12          # 1200÷100
    基价: 320.50
    人工费: 45.00
    材料费: 12.50
    机械费: 263.00
    合价: 3846.00
    换算说明: []
    匹配理由: "清单与定额工作内容一致，机械平整场地"

  - 行标识: 定
    序号: 1.2
    清单编码: "A1-2"
    项目名称: 平整场地 人工
    单位: 100m²
    工程量: 12
    ...
    匹配理由: "辅助人工配合机械"

  - 行标识: 定(补)
    序号: 10.1
    清单编码: "补-001"
    项目名称: 特殊防护
    单位: 项
    工程量:        # 空，换算不了
    基价:
    匹配理由: "定额库无对应项目，需人工核价"
    高亮: true
```

**关键规则**
- 每行清单下插入空白行套定额
- 定额行标识"定"
- 补充定额标识"定(补)"并高亮
- 多对多：一个清单可套多个定额
- 材料价本期不做

| 施工合同 | 付款条款、变更条款、违约责任、索赔条款 |

| 投标文件 | 不平衡报价、漏项、响应性、偏差表 |

**输出 - PDF《文件审查报告》**
```

审查人：文件审查员

一、合同审查

- 风险等级：高

- 建议：重点优化报价策略

```

### 【后勤部】

**基本信息**
- 层级: 跨域
- 原名: CrossDomainCoordinator
- 性质: 全局协调

**核心职责**
1. 协调各部门工作
2. 处理部门间冲突
3. 判断是否需要人工介入
4. 汇总各部门结果

**工作流程**
```
接收CEO派发的多部门任务
    ↓
并行启动各部门
    ↓
收集各部门结果
    ↓
检查：
  - 任一结果置信度<70%？
  - 全局置信度<70%？
  - 部门间结果矛盾？
    ↓
如有问题 → 标记"需人工介入"
如无问题 → 汇总结果返回
```

**触发人工介入条件**
| 条件 | 阈值 |
|------|------|
| 单个任务置信度 | < 70% |
| 全局平均置信度 | < 70% |
| 部门间结果冲突 | 存在矛盾 |

**输出格式**
```json
{
    "success": true,
    "results": {
        "工程部": {...},
        "信息部": {...},
        "资料部": {...}
    },
    "global_confidence": 82,
    "requires_human_review": false,
    "low_confidence_tasks": [],
    "summary": "各部门任务完成，整体置信度82%"
}
```

## 三、完整协作流程

### 场景1：新项目套定额

```
用户: "启动学校综合楼项目，帮我套一下定额"

首辅
├─ 识别为业务请求
└─ 输出: {mode: business, raw_request: "启动学校综合楼项目，帮我套一下定额"}
    ↓
CEO
├─ 检测是新项目
├─ 创建 /projects/学校综合楼/
├─ 初始化 PROJECT-STATUS.yaml
├─ 记录 timeline: 项目设立
├─ 识别需要工程部
├─ 分发任务给工程部经理
└─ 记录 timeline: 任务分发
    ↓
工程部经理
├─ 启动图纸专员 + 信息专员（并行）
├─ 等待两份PDF
├─ 启动清单整理员
├─ 等待标准化清单
├─ 启动套定额员
├─ 等待定额结果
├─ 复核结果
└─ 返回给CEO
    ↓
CEO
├─ 接收工程部结果
├─ 更新项目状态
├─ 记录 timeline: 工程部完成
└─ 返回给首辅
    ↓
首辅
└─ 格式化输出给用户
```

### 场景2：查询招标信息

```
用户: "查一下最近学校的招标信息"

首辅 → CEO → 识别需要信息部 → 信息部经理
    ↓
信息部经理 → 信息查询员
    ↓
返回结果 → CEO → 首辅 → 用户
```

### 场景3：多部门协作

```
用户: "分析一下这个项目，顺便查一下材料价格"

首辅 → CEO
    ↓
识别需要工程部 + 信息部
    ↓
CEO → 后勤部（协调）
    ↓
后勤部并行启动：
    ├─ 工程部经理（套定额）
    └─ 信息部经理（查价格）
    ↓
后勤部汇总结果
    ↓
返回CEO → 首辅 → 用户
```

## 四、数据流转格式汇总

| 阶段 | 数据格式 |
|------|---------|
| 用户 → 首辅 | 自然语言文本 |
| 首辅 → CEO | JSON {mode, raw_request, keywords, is_followup} |
| CEO → 部门经理 | 项目上下文 + 任务描述 |
| 部门经理 → 专员 | 部门内部格式 |
| 专员 → 部门经理 | 结果 + 置信度 |
| 部门经理 → CEO | JSON {success, content, confidence, metadata} |
| 多部门 → 后勤部 | 各部门结果字典 |
| 后勤部 → CEO | 汇总结果 + 全局置信度 |
| CEO → 首辅 | 项目状态更新 + 结果 |
| 首辅 → 用户 | 格式化文本 |

## 五、异常处理汇总

| 层级 | 异常场景 | 处理方式 |
|------|---------|---------|
| 首辅 | 无法识别意图 | 默认走业务模式 |
| 首辅 | 待办存储失败 | 提示用户，不阻断 |
| CEO | 识别不出部门 | 默认工程部，标记待确认 |
| CEO | 信息不足 | 主动询问用户 |
| CEO | 部门返回失败 | 记录原因，尝试重试 |
| 工程部经理 | 某专员失败 | 记录问题，尝试替代方案 |
| 套定额员 | 无对应定额 | 标记补充定额，高亮 |
| 后勤部 | 部门结果冲突 | 标记需人工介入 |
